# Avaya CDR Journal

Web-система для сбора, обработки и просмотра CDR-записей Avaya.

Проект предназначен для эксплуатации рядом с Avaya Communication Manager и Session Manager: принимать CDR, сохранять сырые записи, собирать цепочки вызовов в понятный журнал и давать администратору быстрый поиск по звонкам.

## Что умеет текущий прототип

- Приём CDR от Avaya Communication Manager по TCP.
- Поддержка custom CDR layout из `change system-parameters cdr`.
- Хранение сырых записей отдельно от итогового журнала.
- Обработка цепочек вызовов по UCID.
- Разбор `cond-code`, длительности, вызывающего и вызываемого номера.
- Справочник транков с понятными названиями направлений.
- Разделение входящих, исходящих, местных, неотвеченных и проблемных вызовов.
- Просмотр сырых CDR для отладки.
- Web-интерфейс для журнала, настроек ACD, словарей и диагностики.
- PostgreSQL в Docker.
- Сервисы под systemd.

## Зачем это нужно

Стандартный CDR от Avaya полезен, но читать его напрямую неудобно:

- один реальный звонок может состоять из нескольких CDR-строк;
- VDN, Vector, Hunt Group, transfer и внутренние участки вызова усложняют картину;
- коды завершения требуют расшифровки;
- направление звонка часто приходится восстанавливать по транкам;
- для диагностики нужно видеть и сырые записи, и итоговый журнал.

Цель проекта - сделать CDR понятным журналом звонков, не теряя сырые данные для проверки.

## Общая схема

```text
Avaya CM
   |
   | TCP CDR
   v
Listener
   |
   v
Raw table
   |
   v
Processor
   |
   v
Processed call journal
   |
   v
Web UI
```

## Почему raw и итоговый журнал разделены

Listener должен быстро принимать поток CDR и не заниматься тяжёлым анализом. Поэтому входящие строки сначала сохраняются как raw. Отдельный обработчик собирает их в итоговые записи журнала.

Это позволяет:

- не терять данные при ошибке обработки;
- повторно обработать raw-записи после изменения логики;
- держать основной журнал чистым и удобным для поиска.

## Статус

Проект находится в активной разработке.

Сейчас основной фокус:

- Communication Manager custom CDR;
- корректная логика по UCID и `cond-code`;
- неотвеченные и проблемные вызовы;
- подготовка поддержки Session Manager CDR XML.
