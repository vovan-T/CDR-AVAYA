# Avaya CDR Journal

Web-система для приёма, обработки и просмотра CDR-записей Avaya.

ПО разворачивается на отдельном Linux-сервере и принимает CDR от Avaya Communication Manager по TCP. Данные сохраняются в PostgreSQL: сырые CDR-строки остаются доступными для отладки, а отдельный обработчик собирает их в итоговый журнал вызовов.

Расчёт на инженера, который уже понимает базовую схему Avaya CM / SM / SBCE, CDR output format, VDN, Vector, Hunt Group и транковую маршрутизацию.

## Что умеет текущий прототип

- Приём CDR от Avaya Communication Manager по TCP.
- Поддержка `customized` CDR layout из `change system-parameters cdr`.
- Контроль длины записи по `Record length`.
- Раздельное хранение raw CDR и итогового журнала.
- Агрегация цепочек вызовов по UCID.
- Анализ `cond-code`, `seq-num`, длительности, calling/dialed number.
- Использование `in-trk-code` и `code-used` для определения направления.
- Справочник транков с внутренними/внешними направлениями.
- Разделение входящих, исходящих, местных, неотвеченных и проблемных записей.
- Просмотр сырых CDR для отладки.
- Web-интерфейс для журнала, настроек ACD, словарей и диагностики.
- PostgreSQL в Docker.
- Сервисы под systemd.

## Зачем это нужно

CDR от Avaya хорошо подходит для биллинга и аудита, но в сыром виде неудобен как оперативный журнал:

- один реальный звонок может состоять из нескольких CDR-строк;
- VDN, Vector, Announcement, Hunt Group и transfer создают промежуточные записи;
- для входящих и исходящих сценариев используются разные поля транков;
- `cond-code` требует контекста всей цепочки UCID;
- для диагностики нужно видеть одновременно raw CDR и итоговый результат обработки.

Цель проекта - получить инженерный журнал вызовов, сохранив возможность проверить каждое решение по исходной CDR-строке.

## Общая схема

```text
Avaya Communication Manager
   |
   | TCP CDR
   v
Listener
   |
   v
Raw table
   |
   v
Processor
   |
   v
Processed call journal
   |
   v
Web UI
```

## Почему raw и итоговый журнал разделены

Listener должен быстро принимать поток CDR и не выполнять тяжёлую логику разбора. Поэтому входящие строки сначала сохраняются в raw-таблицу. Отдельный processor анализирует новые raw-записи и формирует итоговый журнал.

Это позволяет:

- не терять данные при ошибке обработки;
- повторно обработать raw после изменения логики;
- держать основной журнал чистым и удобным для поиска.

## Статус

Проект находится в активной разработке.

Сейчас основной фокус:

- Communication Manager custom CDR;
- корректная логика по UCID и `cond-code`;
- корректное определение направления через `in-trk-code` / `code-used`;
- неотвеченные и проблемные вызовы;
- подготовка поддержки Session Manager CDR XML.
